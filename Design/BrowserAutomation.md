# Design: Browser Automation

## 1. Overview

This document outlines the design for the Browser Automation feature, which is the component responsible for actually applying for jobs. It acts as the "hands" of the system, using a browser automation agent to navigate to job application forms, fill them out with the user's information, and submit them. This is one of the most complex and critical parts of the platform.

## 2. Core Features

### 2.1. Automated Application Processing

-   **Queue Processing:** The agent will process jobs that are in the `Backlog` swimlane on the Roadmap board.
-   **End-to-End Automation:** The goal is to handle the entire application process from start to finish without user intervention.

### 2.2. Intelligent Form Filling

-   **LLM-Powered Mapping:** The agent will use an LLM (via Copilot Kit) to intelligently map the user's profile data to the fields on a job application form. Instead of relying on brittle selectors, it will understand the form semantically (e.g., find the field labeled "First Name" and fill it with the user's first name).
-   **Resume Uploading:** The agent will upload the tailored resume generated by the `ResumeGeneration` service.

### 2.3. Error Handling and Fallback

-   **Action Required:** If the agent encounters a situation it cannot handle (e.g., an unusual form field, a CAPTCHA, or missing information in the user's profile), it will not attempt to guess. Instead, it will stop processing the application, move it to the `Action Required` swimlane, and leave a note explaining the problem. This allows the user to manually complete the application and update their profile if necessary.

## 3. Technical Design

### 3.1. Core Technologies

-   **`agent-browser`:** This library will be used to control a headless browser instance. It provides the low-level capabilities to navigate, read page content, and interact with form elements.
-   **Copilot Kit:** This will be used to integrate with an LLM. The LLM will act as the "brain" for the browser agent, deciding what to do based on the current state of the web page.

### 3.2. Backend Services & Logic

-   **Service File:** `src/lib/services/browserAgent.ts` will contain the core logic for the automation agent.
-   **Trigger:** The process will be initiated by a scheduler or a manual trigger that decides to process items from the `Backlog`.

### 3.3. Application Automation Flow

1.  **Select Application:** The agent picks a job application from the `Backlog`.
2.  **Generate Resume:** It calls the `resumeService` to generate a tailored resume for this specific job. The service returns the file path to the generated PDF.
3.  **Launch Browser:** The agent launches a headless browser using `agent-browser` and navigates to the job application URL.
4.  **Analyze Page:** The agent gets the HTML content of the page and sends it to the LLM, along with the user's complete profile data.
5.  **LLM-Driven Interaction Loop:** The core of the process is a loop driven by the LLM:
    a.  **Prompt:** The prompt to the LLM will be something like: *"You are a job application assistant. Here is the HTML of a web page and the user's profile. Your goal is to fill out the application form. What is the next action to take? Your available actions are: `fill(field_selector, value)`, `select(field_selector, option_text)`, `upload(file_input_selector, file_path)`, `click(button_selector)`, `finish()`, or `error(reason)`. Provide the action and its parameters as a JSON object."*
    b.  **Action Execution:** The agent receives the JSON object from the LLM and executes the specified action using `agent-browser`. For example, if the LLM returns `{ "action": "fill", "selector": "#first_name", "value": "John" }`, the agent will find the element with the ID `first_name` and type "John" into it.
    c.  **State Update:** After the action is performed, the agent gets the updated page state and sends it back to the LLM for the next instruction.
    d.  **Loop:** This continues until the LLM determines the form is complete and returns the `finish()` action, or it encounters a problem and returns `error(reason)`.
6.  **Submit Application:** The final action is typically to `click()` the "Submit" button.
7.  **Update Status:**
    -   If the process completes successfully, the agent calls the `applicationService` to move the application to the `Applied` swimlane.
    -   If the process fails or the LLM calls `error()`, the agent moves the application to the `Action Required` swimlane and records the reason for the failure.

### 3.4. API Endpoints

-   This feature is a background process and does not require its own public API endpoints. It will be triggered by an internal scheduling mechanism and will use the services of other features (applications, profiles, resumes).

## 4. Implementation Notes

-   **Complexity:** This is the most complex feature of the entire system. The interaction between the browser, the LLM, and the application logic is non-trivial.
-   **LLM Prompt Engineering:** The success of the form-filling process is almost entirely dependent on the quality and robustness of the LLM prompt. It needs to be very clear and handle a wide variety of form layouts and edge cases.
-   **Handling CAPTCHAs:** The agent will not be able to solve CAPTCHAs. When a CAPTCHA is detected, the agent should immediately stop and move the application to `Action Required`.
-   **Website Variations:** Every job application form is different. The LLM-based approach is designed to be more resilient than hard-coded selectors, but it will still face challenges with unconventional or poorly designed forms.
-   **State:** The `browserAgent.ts` service is currently a placeholder and needs to be fully implemented. This will be a major development effort.
